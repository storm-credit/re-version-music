# Re:Version Orchestra Plan

Re:Version Music의 오케스트라는 직접 모든 산출물을 쓰는 제작자가 아니라, 각 전문가의 판단을 모아 `같은 감정, 다른 버전`이라는 채널 약속을 지키는 총괄 디렉터다.

## 핵심 결론

OddEngine식 오케스트라 구조는 Re:Version Music에 잘 맞는다. 다만 OddEngine이 `캐릭터와 세계관을 반복하는 MV-first 스튜디오`에 가깝다면, Re:Version Music은 `감정 변환 문법을 반복하는 음악-first 스튜디오`로 운영해야 한다.

- 캐릭터는 곡마다 달라질 수 있다.
- 감정 원형, 변환 규칙, 후렴 훅, 저작권 안전선은 매번 고정 절차로 통과한다.
- 팬픽OST형만 캐릭터/세계관 일관성을 강하게 본다.
- 번안형과 답가형은 캐릭터보다 `시점 전환`과 `문화 감정 번역`이 브랜드다.

## 오케스트라가 직접 소유하는 것

1. 목표 정의
2. 범위 자르기
3. 전문가 배정
4. 승인/보류 판단
5. 최종 산출물 통합
6. 다음 액션 기록

오케스트라는 가사, Suno 프롬프트, 영상 프롬프트를 전부 혼자 만들지 않는다. 대신 각 단계의 산출물이 같은 감정 약속을 가리키는지 확인하고, 위험한 방향을 조기에 막는다.

## 운영 원칙

### 1. 감정이 원곡보다 먼저다

원곡명이나 IP는 훅을 설명하기 위한 입구다. 실제 제작 기준은 원곡의 멜로디, 가사, 장면이 아니라 아래 네 가지다.

- 화자
- 상대
- 사건
- 핵심 감정

### 2. 변환은 한 번에 하나만 크게 바꾼다

한 곡에서 언어, 문화, 시점, 장르를 모두 크게 바꾸면 훅이 흐려진다. 메인 변환축 1개와 보조 변환축 1개까지만 잡는다.

예시:
- 메인: 원곡 화자의 상대방 시점
- 보조: R&B에서 한국 새벽 발라드로 장르 변환

### 3. 후렴이 먼저 승인되어야 한다

쇼츠 채널에서는 후렴이 곡의 제품명이다. 1절과 브릿지는 후렴이 승인된 뒤 확장한다.

승인 기준:
- 한 문장으로 감정이 이해된다.
- 자막만 봐도 상황이 잡힌다.
- 원곡 가사나 멜로디를 떠올리게 하지 않는다.
- 댓글을 부를 만한 질문이 남는다.

### 4. 캐릭터 일관성은 유형별로 다르게 본다

| 유형 | 캐릭터 일관성 | 고정해야 할 것 |
|---|---:|---|
| 번안형 | 낮음 | 문화 변환 규칙, 감정 대비 |
| 답가형 | 중간 | 원곡-답가 관계, 시점 논리 |
| 팬픽OST형 | 높음 | 아키타입, 관계성, 세계관 톤 |
| 오리지널형 | 선택 | 상황 훅, 감정 선명도 |

### 5. 저작권 리뷰는 마지막이 아니라 초반 게이트다

곡 기획 단계에서 이미 안전선을 확인한다. 완성 후 고치면 멜로디, 가사, 제목, 썸네일을 모두 갈아엎게 된다.

## 오케스트라 훅

### Intake Hook

요청을 아래 여섯 항목으로 정리한다.

- 목표
- 유형
- 영감 원천
- 변환 방향
- 필요한 산출물
- 위험 요소

### Context Hook

기존 문서와 비교해 중복, 충돌, 캐릭터/IP 위험을 확인한다.

확인 위치:
- `01. 채널 전략/`
- `02. 곡 기획/`
- `06. 템플릿/`
- `결과물/*/metadata/`

### Assignment Hook

이번 작업에 필요한 전문가만 배정한다. 모든 전문가를 매번 호출하지 않는다.

예시:
- 곡 아이디어 검토: 감정 원형 분석가 + 변환 전략가 + 저작권 리뷰어
- Suno 제작: 후렴/훅 작가 + Suno 음악감독
- 영상 패키징: 비주얼 감독 + 발행/성과 분석가

### Review Hook

아래 세 문장이 모두 같은 방향이면 승인한다.

- 제목이 약속하는 감정
- 후렴이 터뜨리는 감정
- 영상이 보여주는 감정

### Publish Hook

최종본은 다음 제작에 다시 쓰일 수 있게 저장한다.

- 곡 기획: `02. 곡 기획/`
- 제작 산출물: `03. 제작/`
- 영상/썸네일: `04. 영상/`
- 발행 기록: `05. 발행/`
- 반복 양식: `06. 템플릿/`

## 추천 제작 흐름

1. 오케스트라 인테이크 작성
2. 감정 원형 추출
3. 변환 규칙 결정
4. 후렴 3안 작성
5. 저작권/감정 일관성 1차 리뷰
6. 가사 확장
7. Suno 프롬프트 작성
8. 영상/Veo 프롬프트 작성
9. 업로드 패키지 작성
10. 발행 후 성과 기록

## 초기 적용 우선순위

1. BTS 답가 시리즈의 저작권 안전선 재점검 — 완료
2. 번안형 10곡의 변환축 단순화 — 완료
3. 팬픽OST형의 캐릭터 아키타입명 정리
4. 곡별 후렴 승인 체크리스트 도입
5. 발행 기록에 `훅/유지율/댓글 반응` 필드 추가

## 오케스트라 승인 문장

작업 종료 시 오케스트라는 아래 형식으로 판단을 남긴다.

```text
승인 상태: 승인 / 조건부 승인 / 보류
감정 원형: ...
메인 변환축: ...
가장 강한 후렴 훅: ...
저작권 위험: 낮음 / 중간 / 높음
다음 액션: ...
```
